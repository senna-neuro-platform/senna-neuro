# ADR-0001: Versioning and Changelog Policy

Status: Accepted  
Date: 2026-03-07

## Context

Нужно единообразно вести версии проекта и журнал изменений, чтобы:
- версия однозначно показывала шаг MVP и итерацию внутри шага;
- правила инкремента были формальными и проверяемыми;
- changelog не расходился по структуре между разными коммитами.

## Decision

1. Формат версии: `A.B.C-dev`.
2. `A` (major): до первого релиза всегда `0`.
3. `B` (minor): равен номеру текущего шага MVP-плана.
4. `C` (patch/build): увеличивается на `+1` при каждом изменении в рамках текущего шага.
5. При переходе на новый шаг MVP: `B = B + 1`, `C = 0`.
6. Единый источник версии: файл `VERSION`.
7. Changelog ведется в `CHANGELOG.md`:
   - блоки группируются по `B` (minor), то есть один блок на шаг MVP;
   - заголовок блока: `## DD.MM.YYYY \`A.B.C-dev\`` (текущая версия шага);
   - при изменениях внутри того же `B`: увеличивается только `C`, версия в заголовке этого же блока обновляется, новые пункты добавляются в начало списка блока;
   - новый блок создается только при переходе на новый `B` и ставится выше предыдущих;
   - внутри блока фиксируются только фактические изменения проекта; служебные записи про "обновили версию/changelog" не добавляются;
   - порядок всегда обратный хронологический: новое сверху.

## Consequences

- Любое изменение в шаге требует обновления `VERSION` и верхнего блока `CHANGELOG.md` для соответствующего `B`.
- Номер шага MVP всегда читается напрямую из `B`.
- История изменений сохраняется в обратном хронологическом порядке.
